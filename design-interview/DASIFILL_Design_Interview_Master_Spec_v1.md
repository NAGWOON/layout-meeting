# DASIFILL Design Interview Web App — Master Spec v1
작성일: 2026-04-13  
목적: Claude 구현 전, 기능/질문/로직/구조를 고정하기 위한 기준 문서

---

## 0. 문서의 역할

이 문서는 **UI 시안 문서가 아니라 기능/로직/콘텐츠 구조 명세서**이다.  
향후 작업 순서는 아래와 같다.

1. 본 문서를 기준으로 기능/질문/로직 구조를 고정
2. UI/UX 비주얼 방향은 별도 Gemini 산출물로 확정
3. Claude에는 본 문서 + UI 가이드를 함께 전달
4. Claude는 디자인을 임의 재해석하지 말고, 본 문서의 기능 로직을 우선 구현

이 문서는 대화 누적에 따른 누락/왜곡을 방지하기 위한 **Master Spec** 이다.

## 1. 제품 정의

### 1-1. 제품명 (임시)
- DASIFILL Design Brief
- DASIFILL Pre-Design Interview
- DASIFILL Client Brief System

최종 네이밍은 추후 확정 가능하나, 본 스펙에서는 `Design Interview` 로 표기한다.

### 1-2. 제품 성격
이 앱은 일반 문의용 설문이 아니다.  
**계약 후 / 현장조사 후 / 1차 디자인 제안 전** 고객의 생활 방식과 설계 요구를 구조적으로 수집하는 **프리미엄 브리프 시스템**이다.

### 1-3. 핵심 목적 우선순위
1. 생활 패턴 / 불편 / 사용 방식 파악
2. 설계에 필요한 핵심 실무 정보 수집
3. 대면 미팅 피로도 및 반복 인터뷰 시간 감소
4. 체계적이고 전문적인 프로세스 경험 제공

### 1-4. 제외 범위
- 디자인 취향 / 무드보드 / 스타일 선호는 별도 앱에서 처리 예정
- 가전제품 상세 선택은 별도 앱/설문에서 처리 예정
- 본 앱에서 과도한 감성 카피/시적인 질문은 지양
- 본 앱은 “감성 표현”보다 “실제 설계 인풋 확보”가 우선

## 2. 사용 맥락

### 2-1. 제공 시점
- 계약 후
- 현장 조사/실측 후
- 1차 디자인 미팅 전 또는 직후
- 3D / 본격 설계안 제안 전

### 2-2. 작성 주체
- 고객 단독 작성 가능
- 부부 공동 작성 가능
- 고객이 함께 논의하며 작성하는 상황을 고려

### 2-3. 목표 소요 시간
- 목표: 15~20분
- 체감 이상적인 목표: 12~15분
- 평균 고객 최대 허용치: 20분 내외

### 2-4. UX 원칙
- 선택형 중심
- 일부 핵심만 자유서술
- “정답 검사”가 아니라 “생각 과정 읽기”가 목적
- 고객이 답하기 어려운 전문용어는 최소화
- 너무 시적이거나 오글거리는 표현 금지
- 디자인 브랜딩은 카피보다 **UI/UX의 퀄리티** 로 전달

---

## 3. 브랜딩/톤 원칙

### 3-1. Tone of Voice
- Professional + Warm
- 감성적 과잉 금지
- 문학적/시적 표현 금지
- 너무 딱딱한 행정 문구도 지양
- 고객이 “체계적이고 생각이 깊다”는 인상을 받되 부담스럽지 않아야 함

### 3-2. 질문 원칙
- 질문은 practical 해야 한다
- 설계 판단에 도움되지 않는 질문은 제거
- 너무 obvious한 질문은 줄인다
- 선택지는 “단어”보다 “맥락이 느껴지는 서술형”이 좋다
- 고객이 “아, 이 회사는 이런 걸 보네”라고 느낄 정도의 세밀함은 유지

### 3-3. 브랜딩 전달 방식
- 감성 문구를 전면에 내세우지 않는다
- 브랜드의 프리미엄감은 UI 디자인, spacing, hierarchy, 구성 방식에서 전달
- 앱은 “업무툴”보다 “전용 인터뷰 시스템”처럼 느껴져야 한다

## 4. 기능 구조 요약

### 4-1. 필수 기능
- 프로젝트 생성/저장
- 진행률 표시
- 섹션/공간별 응답 저장
- branch logic
- repeatable section logic
- 요약/정리 출력
- 기존 데이터 보존
- 고객별 저장 및 재진입 가능
- 선택형 중심 UI
- “아직 정해지지 않음 / 제안받고 싶음” 선택지 적극 제공

### 4-2. 저장 원칙
- 기존 앱(localStorage 포함)과 namespace 분리
- design-interview 전용 storage key 사용
- 기존 layout-meeting 데이터 절대 덮어쓰지 말 것

### 4-3. 예상 UI 기능 (디자인 구현은 별도 문서)
- 상단 progress bar
- 섹션 전환
- branch에 따른 질문 노출/숨김
- repeatable 욕실 섹션 생성
- 마지막 summary/export
- “저장됨” 상태 표시
- 고객이 현재 몇 단계에 와 있는지 명확히 보여줄 것

### 4-4. 섹션 전환 안내 문구
섹션 전환 시 짧은 안내 문구를 넣어 UX 텐션을 부드럽게 한다.

예시:
- “이제 실제 공간 계획을 위해 공간별 사용 방식과 필요 요소를 확인하겠습니다.”
- “마지막으로 추가 요청사항을 확인하겠습니다.”

포함 공간:

### PART 5. 최종 요청사항

## 6. 공통 UX 규칙

### 6-1. 선택형 우선
대부분 single/multi-choice로 처리

### 6-2. 자유서술 최소화
자유서술은 마지막 기타/최종 요청에 집중
파트마다 자유서술 삽입 금지

### 6-3. Escape Option 유지
가능한 질문에 아래 옵션 유지:
- 아직 정해지지 않음
- 제안받고 싶음

### 6-4. Branching 최소화 원칙
branch는 wow 포인트가 되는 핵심 파트에만 적용
- 거실: 적극 적용
- 주방: 제한적 적용
- 욕실: 계층형 적용
- 나머지: 기본 정적 구조 유지

### 6-5. 정량 질문 원칙
정량 질문은 아래 공간에만 제한적으로 적용:
- 현관
- 드레스룸
- 주방은 정량보다 방향/의도/디테일 중심으로 간다

### 6-6. 비용 옵션 표기 원칙
비용 차가 큰 옵션은 전 공간이 아니라 **욕실 중심**으로만 표시

비용표기 단어:
- 기본 사양
- 업그레이드 옵션
- 프리미엄 옵션

“고비용” 같은 부정적 단어는 사용하지 않는다.

## 7. PART 1 상세 명세 — 현재의 삶 이해

### 목적
- 가족 구성
- 집 체류 패턴
- 주말 사용 방식
- 가족이 모이는 방식
- 집의 역할 인식

### 문항

#### Q1. 현재 함께 거주하는 구성원은 어떻게 되시나요?
- 부부
- 자녀 1명
- 자녀 2명 이상
- 부모님 동거
- 반려동물
- 기타

#### Q2. 평소 집에 머무르는 시간은 어느 정도인가요?
- 하루 대부분 집에서 생활합니다
- 주로 저녁/주말에 머뭅니다
- 생활보다 휴식/수면 위주로 이용합니다
- 구성원마다 생활 패턴이 다릅니다

#### Q3. 주말에는 주로 어떤 방식으로 시간을 보내시나요? (복수 선택)
- 집에서 충분히 쉬며 보내는 편입니다
- 외출 / 외부 활동이 많은 편입니다
- 가족과 함께 시간을 보내는 편입니다
- 집안 정리 / 청소 / 관리에 시간을 씁니다
- 취미 / 자기계발 활동을 합니다
- 손님을 초대하거나 모임을 갖는 편입니다

#### Q4. 가족이 함께 있을 때 가장 자주 머무는 공간은 어디인가요?
- 거실에 자연스럽게 모이는 편입니다
- 주방/식탁에서 함께 보내는 시간이 많습니다
- 각자 방이나 개인 공간에서 보내는 편입니다
- 상황에 따라 유동적으로 달라집니다

#### Q5. 집에서 가장 자주 이루어지는 활동은 무엇인가요? (복수 선택)
- 휴식 / 미디어 시청 중심
- 식사 / 요리 / 다과 중심
- 업무 / 학습 / 집중 활동
- 취미 / 자기계발 / 독서
- 육아 / 놀이 / 가족 활동
- 손님 초대 / 교류 / 모임

#### Q6. 당신에게 집은 어떤 역할에 더 가까운 공간인가요?
- 하루의 피로를 풀고 쉬는 공간
- 가족과 시간을 보내는 중심 공간
- 혼자만의 루틴과 시간을 보내는 공간
- 일과 생활이 함께 이루어지는 공간
- 사람들과 관계를 나누는 공간

## 8. PART 2 상세 명세 — 설계 우선순위 / 가치 기준

### 목적
공간별 니즈와 겹치지 않도록, “무엇을 원하느냐”가 아니라 “무엇을 우선시하느냐”를 묻는다.

### 문항

#### Q1. 공간을 계획할 때 가장 중요하게 생각하는 요소는 무엇인가요?
- 공간이 넓고 개방감 있게 느껴지는 것
- 생활하기 편한 동선과 구조
- 충분한 수납과 정리의 용이함
- 깔끔하고 정돈된 분위기
- 유지관리 및 청소의 편리함
- 가족 간 소통과 교류

#### Q2. 디자인과 공간 구성 중 어느 쪽에 더 비중을 두고 싶으신가요?
- 분위기/디자인 완성도가 더 중요합니다
- 실용성과 사용 편의가 더 중요합니다
- 두 요소의 균형이 중요합니다

#### Q3. 공간 계획 시 어떤 방향을 더 선호하시나요?
- 현재 생활 방식에 최적화된 구성
- 향후 변화까지 고려한 유연한 구성
- 누구나 편하게 사용할 수 있는 안정적인 구성

#### Q4. 이번 프로젝트를 통해 가장 기대하는 변화는 무엇인가요?
- 생활 만족도가 높아지는 것
- 공간 활용도가 좋아지는 것
- 전체 분위기/완성도가 좋아지는 것
- 관리와 유지가 편해지는 것
- 가족 간 생활 방식이 개선되는 것

## 9. PART 3 상세 명세 — 미래 고려사항

### 목적
설계에 직접적인 영향은 제한적일 수 있으나, 고객에게 “우리는 현재뿐 아니라 앞으로의 변화까지 고려합니다”라는 전문적 인상을 준다.

### 문항

#### Q1. 향후 3~5년 내 가족 구성 변화 계획이 있으신가요?
- 자녀 계획이 있습니다
- 자녀의 성장/독립 예정이 있습니다
- 가족/부모님 동거 가능성이 있습니다
- 특별한 변화 계획은 없습니다

#### Q2. 향후 생활 방식 변화 가능성이 있으신가요?
- 재택/홈오피스 필요 가능성이 있습니다
- 취미/작업 공간이 필요해질 수 있습니다
- 수납/물품 증가가 예상됩니다
- 특별한 변화는 없을 것 같습니다

#### Q3. 이번 공간은 어느 정도 기간 사용 예정이신가요?
- 단기 거주 예정 (5년 이하)
- 중기 거주 예정 (5~10년)
- 장기 실거주 예정

- 벽걸이/심플한 구성
- 매립/반매립형 연출

- 명칭 라벨
- 상세/간단 모드 제공 권장

## 13. 데이터 구조 설계 원칙

### 13-1. 질문 구조
각 질문은 최소 아래 필드를 가져야 함:
- id
- section
- label
- type
- options
- is_required
- branch_condition (optional)
- repeatable (optional)
- helper_text (optional)
- cost_tier_note (optional)

### 13-2. branch_condition 예시
- if `living.center_focus == "tv"`
- if `kitchen.island_type in ["bar_island", "dining_island"]`
- if `bathroom.vanity_type == "counter"`

### 13-3. repeatable 예시
- bathroom: true
- bathroom_count 기반 생성

### 13-4. 결과 저장 형식
- 프로젝트별 저장
- 섹션별 응답
- 반복 섹션은 배열 구조
- summary 출력 시 공간명 기준 정리

예:
```json
{
  "bathrooms": [
    {
      "label": "공용욕실",
      "mood": "...",
      "use_type": "...",
      "bathtub": "...",
      "shower_type": "...",
      "vanity_type": "...",
      "counter_sink_type": "..."
    }
  ]
}
```
